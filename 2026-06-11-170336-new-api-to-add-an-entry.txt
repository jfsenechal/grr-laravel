  What I added

  - app/Http/Requests/Api/StoreEntryRequest.php ‚ÄĒ Form Request validating
  room_id (must exist in grr_room), name, start_time/end_time (unix timestamps,

  - EntryController::store() ‚ÄĒ creates the entry, stamps create_by from the

  - Route registered in routes/api.php alongside the existing GET /entries.
  - 5 new tests in tests/Feature/Api/EntryApiTest.php (auth required, happy
  path, required-field validation, end-before-start, unknown room).

  Two pre-existing bugs I had to fix to make tests pass

  1. Wrong API prefix in the existing tests ‚ÄĒ they called /api/entries, but
  bootstrap/app.php sets apiPrefix: 'esquare'. The 5 existing index tests were
  silently failing (404). Updated them to /esquare/entries.
  2. Migration column names out of sync with the model/DB ‚ÄĒ create_entries_table
  used English names (status_entry, days, key, mail, max_participant_count,
  deleted) while the Entry model, EntryResource, and the real database use the
  legacy French names (statut_entry, jours, clef, courrier, nbparticipantmax,
  supprimer). Under strict-model mode the resource threw on the missing
  statut_entry attribute. Renamed those 6 columns in the migration to match.

  All 10 tests pass; Pint and PHPStan (level 5) are clean on the changed files.

  A note on #2: I edited an existing migration in place. That's only safe if the
  test/dev databases get re-migrated (which RefreshDatabase handles). If this
  migration has already run in a production database, renaming columns there
  needs a separate ALTER migration instead ‚ÄĒ let me know if that applies and
  I'll write one.

  One design choice worth confirming: the API takes start_time/end_time as
  integer unix timestamps (symmetric with what GET /entries returns). If you'd
  rather accept ISO date strings (e.g. 2026-03-23T08:00) and convert internally,
  I can adjust the request.
